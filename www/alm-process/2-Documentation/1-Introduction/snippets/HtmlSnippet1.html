Contribute

Development in the M3X project is collaborative with the ultimate goal of providing tools and APIs for all JSR184 and JSR297 users. This lofty goal can be reached if users contribute code and ideas to the project. This may seem daunting, but even simple feedback like a thank you will help. Code patches and build script improvements are always welcome.

Please note that patches may not always be approved. The core developers will do their best to explain in a concise manner the reasons for rejection. Though depending on the nature of the patch that might not always be possible. Nevertheless your help is appreciated

    There are a few ways you can contribute:
  • Submit a bug report in the Issue Tracker
  • Subscribe to one or many of the mailing lists and have your say.
      Choose the ones that most suit your purpose:
    • dev@m3x.dev.java.net for patch committers and core developers.
    • issues@m3x.dev.java.net for following the issues in the tracker.
    • ecosystem@m3x.dev.java.net for discussing the broader scope of M3X, JSR184 and JSR297
  • Join the project and edit the source directly. Commit privileges may not be granted until a user has contributed through other means. If this is your desire, please submit your changes as patches first. Such rights will only be granted by the current Benevolent Dictator.

Best Practises

Coding Standards

When working with the source code always follow the coding standard. This is a requirement not an option. The standard is there to ensure all developers experience a uniform look to the source code. The coding standard is enforced in the build script using the excellent Checkstyle tool. Style errors are regarded as errors in the code and should not be committed to the project unless in special circumstances (auto-generated code may sometimes violate this in an acceptable manner).

Unit Testing

Wherever possible automated Unit Testing should be used. The M3X project already has quite an extensive suite of tests, but we always need more. Patches submitted with a corresponding test will always take precedence to patches that require manual testing.

For all the JVM based code JUnit 3+ compatible tests should be used. The build script runs JUnit in all the sub-projects and any new tests should be added to those preexisting test suites if applicable.
